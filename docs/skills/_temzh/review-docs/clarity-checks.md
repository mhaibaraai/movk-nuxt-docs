# 清晰度审查指南

评估 Docus 文档中内容清晰度和可读性的指南。

## 语气和声调

### 主动语态
✅ 好的："通过添加...来配置你的应用"
❌ 避免："应用可以通过添加...来配置"

### 现在时
✅ 好的："服务器处理请求..."
❌ 避免："服务器将处理请求..."

### 第二人称
✅ 好的："你可以自定义主题..."
❌ 避免："用户可以自定义主题..."

## 句子结构

**目标：** 每句 15-20 字
**最大：** 在标记审查前 25 字

示例：
- 太长（32 字）："当你想配置你的应用以使用具有 OAuth 提供商的身份验证时，你应该首先安装所需的依赖项，然后配置你的环境变量。"
- 更好（分成 2 句）："首先，为 OAuth 安装所需的依赖项。然后，配置你的身份验证环境变量。"

## 段落结构

**指南：**
- 2-5 句每个段落
- 每个段落一个主要思想
- 标题之间 200-400 字

## 基于动作的标题

### 最佳实践：遵循 Nuxt 模式

现代文档（如 [Nuxt](https://nuxt.com/docs)）在指南和教程页面中始终使用 **动作动词**。这使文档易读且任务导向。

### 何时使用动作动词

**页面标题（H1）** - 用于：
- 指南页面："创建你的第一个模块"、"构建自定义插件"
- 教程页面："部署你的应用"、"设置身份验证"
- 操作方法页面："添加深色模式"、"配置环境变量"
- 食谱页面："实现实时更新"

**标题（H2/H3）** - 用于：
- 指南和教程中的所有步骤
- 顺序操作
- 配置部分
- 实现细节

### 何时使用名词（例外）

- **文件名：** 始终保持为名词/kebab-case（`1.introduction.md`、`2.installation.md`）
- **开始使用标题：** "介绍"、"安装"、"快速开始"（方向页面）
- **API 参考页面：** 使用函数/组件名称（"useSession"、"Button"、"defineNuxtConfig"）
- **概念页面：** 在解释理论时可以使用名词（"架构"、"核心概念"、"最佳实践"）

### 动作动词示例

| 类别 | 动词 |
|----------|-------|
| 主要 | 创建、构建、使用、开发、运行、发布、添加、配置、设置、启用、连接 |
| 次要 | 处理、自定义、部署、实现、集成、安装、定义、编写、测试、调试、更新 |

### 真实世界示例（Nuxt 模式）

**页面标题（H1）：**
```markdown
# 创建你的第一个模块
```

**部分标题（H2）：**
```markdown
## 创建模块
## 使用启动模板
## 开发你的模块
```

**小部分标题（H3）：**
```markdown
### 运行测试
### 构建你的模块
### 发布到 npm
```

### 坏 vs 好的示例

| 静态/名词（避免） | 基于动作（使用） |
|---------------------|-------------------|
| 配置 | 配置你的应用 |
| 模块创建 | 创建新模块 |
| 数据库设置 | 连接数据库 |
| 路由保护 | 保护你的路由 |
| 会话管理 | 处理用户会话 |
| 错误处理 | 优雅地处理错误 |
| 测试 | 运行你的测试 |
| 部署 | 部署到生产 |

## 术语一致性

**检查：**
- 在"config"和"configuration"之间交替
- 在"app"和"application"之间切换
- 大小写不一致（"nuxt"vs"Nuxt"）
- 对同一概念使用混合的术语

**验证：**
- 技术术语在首次使用时定义
- 术语命名一致
- 产品特定的术语系统使用

## 代码示例

### 质量清单
- [ ] 完整且可复制粘贴（不是片段）
- [ ] 代表文件的所有代码块上的文件名标签（例如，` ```vue [App.vue] `、` ```ts [server.ts] `、` ```tsx [App.tsx] `）
- [ ] 代码语言匹配项目的栈（例如，如果项目使用的是 TypeScript）
- [ ] 注释解释非显而易见的逻辑
- [ ] 现实的变量名称（不是 foo/bar）
- [ ] 工作代码（无占位符值如 `YOUR_API_KEY`）
- [ ] 一致的缩进和风格

### 多包管理器支持

始终为覆盖 **项目/生态系统支持的所有包管理器** 的安装命令使用 `::code-group`：

```markdown
::code-group
```bash [pnpm]
pnpm add package-name
```

```bash [npm]
npm install package-name
```

```bash [yarn]
yarn add package-name
```

```bash [bun]
bun add package-name
```
::
```

检查项目的 README、lock 文件或现有文档，以确定要包含哪些包管理器。确保没有丢失——常见的被遗忘是较新的。

### 显示呈现输出

使用 `::code-preview` 在源代码旁边显示呈现的结果。这对于记录视觉功能（如 markdown 语法、代码块元数据、组件呈现）非常理想：

```markdown
::code-preview
- 任务 1
- [x] 任务 2（已完成）

#code
```mdc
- 任务 1
- [x] 任务 2（已完成）
```
::
```

### 代码组最佳实践

**为以下内容使用 `::code-group`：**
- 包管理器安装变量（pnpm/npm/yarn/bun）
- 框架变体（Vue / React 并排）
- 代码 + 输出对
- 语法 + AST 对

**不要使用 `::code-group` 来混合无关的步骤：**
- 终端安装命令和配置文件编辑——这些是顺序操作，而不是等效替有品体。将它们保持为单独的块，它们之间有过渡文本。

## 常见的清晰度问题

### 说明中的被动语态
❌ "文件应该在根目录中创建"
✅ "在根目录中创建文件"

### 冗长短语
| 冗长 | 简洁 |
|--------|---------|
| 为了 | 到 |
| 重要的是要注意 | 注意： |
| 到目前为止 | 现在 |
| 由于这个事实 | 因为 |

### 模糊代词
❌ "这允许你..."
✅ "此配置允许你..."

### 未解释的行话
❌ "使用 SSR 以获得更好的性能"
✅ "使用服务器端呈现（SSR）以获得更好的性能"
